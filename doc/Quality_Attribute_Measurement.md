# 품질 요소 측정

## 개요

본 문서는 WorkDrive(클라우드 파일 공유 시스템)에 대해 사용자·개발(유지보수)·관리자 관점에서 기대하는 품질 요소를 정리하고, 측정 가능한 목표와 기능 간 매핑, 측정 방법을 제시한다. 이후 프로젝트 관리 계획 수립 시 품질 목표로 반영할 수 있도록 한다.

## 1) 사용자가 원하는 기대치

- **사용성(Usability):** 주요 작업(업로드/공유/검색/파일 댓글 확인·작성) 완료까지 평균 클릭 수 5회 이하를 목표로 한다.
- **성능(Performance):** 파일 검색과 다운로드 결과가 빠르게 제공되어 업무 흐름이 끊기지 않아야 한다.
- **가용성(Availability):** 언제 어디서나 접속 가능하고, 주요 시간대에도 안정적으로 서비스가 유지되어야 한다.
- **신뢰성(Reliability):** 파일 손실 없이 저장되고 버전 이력이 정확히 관리되어 이전 상태 복구가 가능해야 한다.
- **보안(Security):** 로그인 인증, 권한 기반 접근 제어(열람·수정·댓글 등 공유 권한 구분 포함), 암호화 전송, 만료 가능한 공유 링크로 데이터 유출 위험을 줄여야 한다.
- **협업성(Collaboration):** 동시 작업 시 충돌 발생률 1% 이하를 목표로 하며, 충돌 발생 시 복구 절차를 제공한다. 파일 단위 댓글로 협업 의견을 남길 수 있어야 하며, 댓글 권한이 없는 사용자는 댓글을 작성할 수 없어야 한다.

## 2) 개발자/유지보수 담당자가 원하는 기대치

- **유지보수성(Maintainability):** 기능 변경 시 영향 파일 수 평균 5개 이하를 목표로 하고, 모듈 경계를 유지한다.
- **확장성(Scalability):** 사용자 수/파일 수 증가(조직 규모 확대, 대용량 파일 증가)에도 성능 저하를 완화할 수 있어야 한다.
- **시험 용이성(Testability):** 권한 정책(열람·수정·댓글 권한), 버전 복원, 검색 필터, 공유 링크 만료 같은 핵심 로직을 자동 테스트로 검증 가능해야 한다.
- **관측 가능성(Observability):** 접근/수정/공유 이벤트 로그를 통해 장애 분석 및 감사 대응이 가능해야 한다.
- **이식성(Portability):** 웹 기반 구조로 브라우저/배포 환경 변화에 대응하기 쉬워야 한다.

## 3) 개발 조직 관리자가 원하는 기대치

- **보안 및 컴플라이언스:** 권한 정책, 접근 기록, 감사 로그를 통해 내부 통제와 책임 추적이 가능해야 한다.
- **운영 안정성:** 조직 약 200명 규모의 동시 사용 상황에서도 핵심 기능 SLA를 만족하도록 운영되어야 한다.
- **비용 효율성:** 저장공간, 트래픽, 운영 인력 비용을 예측 가능하게 관리할 수 있어야 한다.
- **확장 로드맵 적합성:** 향후 문서 공동 편집, AI 검색 등 확장 요구를 수용할 수 있는 구조여야 한다.
- **리스크 관리:** 외부 공유 시 만료 링크/권한 제한/접속 기록으로 정보 유출 리스크를 통제할 수 있어야 한다.

## 우선순위 품질 요소

1. **보안(Security)**
2. **성능(Performance)**
3. **신뢰성(Reliability)**
4. **유지보수성(Maintainability)**
5. **사용성(Usability)**

위 우선순위는 요구사항의 핵심 문제(버전 혼선, 파일 탐색 지연, 외부 공유 보안)를 먼저 해결하기 위한 기준이다.

### 순위 배정 근거

아래는 동일 도메인(조직용 클라우드 파일 공유)에서 **왜 이 순서로 우선순위를 두었는지**에 대한 설명이다. 수치 측정은 아직 수행하지 않았으나, 품질 목표를 논의할 때 순위의 의미를 공유하기 위한 근거로 쓴다.

| 순위 | 품질 요소 | 배정 이유 |
|:----:|-----------|-----------|
| 1 | **보안** | 조직 자료·개인정보 유출, 권한 오설정으로 인한 비인가 접근은 **법적·평판적 피해**로 직결된다. 성능이나 UI를 개선해도 보안 사고 한 건이 프로젝트 신뢰를 상실시킬 수 있어 **최우선**으로 둔다. |
| 2 | **성능** | 검색·목록·다운로드가 느리면 업무가 **중단**되고 사용자가 시스템을 우회하게 된다. 보안 요구를 만족한 뒤, **일상 업무 체감**에 가장 크게 영향을 주는 요소로 2순위로 둔다. |
| 3 | **신뢰성** | 파일 손실, 잘못된 버전 복원, 동시 수정으로 인한 데이터 깨짐은 **복구 비용과 업무 혼선**이 크다. 성능보다 “느리지만 맞다”가 우선되는 영역은 아니지만, **데이터 무결성**은 조직 저장소의 전제이므로 성능 직후에 둔다. |
| 4 | **유지보수성** | 기능 추가·결함 수정 시 변경 범위가 넓어지면 보안·성능 개선도 **지연**된다. 장기적으로 품질을 지속하려면 모듈 경계와 테스트 가능성을 초기부터 잡아야 하므로 4순위로 둔다. |
| 5 | **사용성** | 클릭 수·화면 이해도 등은 매우 중요하나, **보안 사고·데이터 손실·심각한 지연**이 없는 상태에서 차별화와 만족도가 두드러진다. 1~3순위가 충족된 뒤에도 **지속적으로 개선**하는 영역으로 두었다. |

> **참고:** 가용성·협업성 등 다른 품질 요소는 위 다섯 가지와 **겹치거나** 별도 지표(가동률, 충돌률 등)로 이미 다루므로, “우선순위 품질 요소” 목록에는 핵심 트레이드오프를 논의하기 쉬운 위 다섯 가지를 올렸다.

## 품질 목표(초안, 측정 기준)

| 품질 요소 | 측정 기준(예시) | 목표(초안) |
|------|------|------|
| **성능(검색)** | 파일명 검색 응답시간 (P95) | 2초 이내 |
| **성능(다운로드)** | 다운로드 요청부터 스트림 시작(TTFB)까지 시간 (P95) | 3초 이내 |
| **가용성** | 월간 서비스 가동률 | 99.5% 이상 |
| **신뢰성** | 파일 업로드 성공률 (네트워크 정상 상태 기준) | 99.9% 이상 |
| **복구성** | 버전 복원 요청 완료 시간 | 1분 이내 |
| **보안(접근통제)** | 비인가 접근 차단율 | 100% (권한 정책 기준) |
| **보안(외부공유)** | 만료 링크 정책 적용률 | 100% |
| **보안(공유·댓글 권한)** | 댓글 권한 없이 댓글 작성이 성공한 비율 | 0% (권한 정책 기준) |
| **협업성(파일 댓글)** | 파일 댓글 목록 조회 응답시간 (P95) | 2초 이내 |
| **감사 가능성** | 파일 접근/수정/댓글 이벤트 로그 누락률 | 0%에 근접(정책상 필수 항목 100% 기록) |
| **관측 가능성** | 로그 수집 지연 시간 | 5초 이하 |

> 위 수치는 설계 초안 기준이며, 프로젝트 진행 중 인프라/테스트 결과에 따라 조정한다.

## 품질 요소-핵심 기능 매핑

| 핵심 기능 | 우선 품질 요소 | 반영 포인트 |
|------|------|------|
| 파일 업로드/다운로드 | 성능, 신뢰성 | 대용량 업로드 안정화, 재시도 처리, 전송 상태 표시 |
| 파일/폴더 공유 | 보안, 사용성 | 공유 대상 명확화, 열람·수정·댓글 권한 구분, 권한 프리셋, 외부 링크 만료 설정 |
| 파일 댓글 | 보안, 협업성, 성능 | 댓글 권한 검증, 스레드·작성자 표시, 댓글 조회·작성 응답 시간 |
| 접근 권한 관리(RBAC) | 보안, 유지보수성 | 역할 템플릿, 권한 변경 이력, 정책 일관성 검증 |
| 파일 버전 관리 | 신뢰성, 복구성 | 버전 목록 조회, 이전 버전 복원, 이력 무결성 보장 |
| 검색 | 성능, 사용성 | 파일명/사용자/날짜 필터, 빠른 결과 반환, 정렬/필터 UX |
| 활동 로그/감사 | 관측 가능성, 컴플라이언스 | 접근/수정/공유/댓글 이벤트 표준화, 감사 조회 기능 |

## 측정 방법

각 품질 지표는 다음 방법으로 측정한다.
- 성능: 서버 로그 및 APM 기반 수집(P95 기준)
- 가용성: 모니터링 시스템의 Uptime 기준 집계
- 보안: 접근 제어 로그 및 감사 기록 분석
- 신뢰성: 요청 성공/실패 로그 기반 계산
- 협업성(댓글): 댓글 API 응답시간·권한 거부 로그 기반 집계
