### 1.2 프로젝트 개요

#### 1.2.1 프로젝트 정의

WorkDrive는 조직 구성원 간 효율적인 협업을 지원하기 위한 클라우드 기반 파일 공유 시스템이다. 사용자는 웹 브라우저에서 파일을 업로드, 다운로드, 검색하고 팀원과 공유할 수 있으며, 역할 기반 접근 제어를 통해 열람, 수정, 댓글, 관리자 권한을 구분하여 안전하게 협업할 수 있다.

파일 버전 관리와 실시간 동기화를 통해 동시 작업 중 발생할 수 있는 버전 혼선과 충돌을 줄이며, 활동 로그와 감사 기록을 통해 조직 내 파일 접근과 변경 이력을 추적할 수 있도록 한다.

#### 1.2.2 주요 기능 설명

| ID | 주요 기능 | 설명 |
|----|-----------|------|
| F01 | 파일 업로드/다운로드 | 문서, 이미지 등 다양한 파일을 중앙 저장소에 업로드하고 필요 시 다운로드한다. |
| F02 | 파일/폴더 공유 | 팀원 또는 지정 사용자에게 파일과 폴더를 공유한다. |
| F03 | 파일 댓글 | 공유된 파일에 협업 의견을 남기고 권한에 따라 댓글 작성 가능 여부를 제어한다. |
| F04 | 접근 권한 관리 | 읽기, 수정, 댓글, 관리자 등 역할 기반 접근 제어를 제공한다. |
| F05 | 파일 버전 관리 | 파일 변경 이력을 보관하고 이전 버전으로 복구할 수 있도록 한다. |
| F06 | 실시간 동기화 | 서버에 반영된 파일, 버전, 공유 상태가 협업자 화면에 짧은 지연으로 반영되도록 한다. |
| F07 | 검색 | 파일명, 사용자, 날짜 등 조건으로 파일을 빠르게 검색한다. |
| F08 | 활동 로그/감사 | 접근, 수정, 공유, 댓글 이벤트를 기록하여 추적과 감사가 가능하도록 한다. |

### 1.3 프로젝트 목표

WorkDrive 프로젝트의 목표는 조직 업무 자료를 안전하고 효율적으로 공유할 수 있는 웹 기반 클라우드 파일 공유 시스템을 설계하고 구현하는 것이다. 사용자는 설치 없이 브라우저에서 파일을 관리하고, 조직의 권한 정책에 맞게 공유 범위를 지정하며, 버전 관리와 댓글을 통해 협업 과정의 혼선을 줄일 수 있어야 한다.

본 프로젝트는 개발 기간과 비용에 특별한 제한을 두지 않으며, 프로젝트 수행에 필요한 인원을 역할별로 충분히 구성할 수 있다고 가정한다. 따라서 기능 구현뿐 아니라 보안, 성능, 신뢰성, 유지보수성, 사용성을 균형 있게 고려하여 안정적인 협업 시스템을 구축하는 것을 목표로 한다.

### 1.4 용어 정의

| 용어 | 설명 |
|------|------|
| 클라우드 저장소 | 사용자의 파일을 서버에 저장하고 네트워크를 통해 접근하도록 제공하는 저장 공간 |
| RBAC | Role-Based Access Control의 약자로, 역할에 따라 권한을 부여하는 접근 제어 방식 |
| 버전 관리 | 파일의 변경 이력을 저장하고 필요 시 이전 상태로 복구하는 기능 |
| 실시간 동기화 | 서버에 반영된 파일, 버전, 공유 상태가 사용자 화면에 새로고침 없이 또는 짧은 지연으로 반영되는 것 |
| 감사 로그 | 누가 언제 어떤 파일에 접근하거나 변경했는지 기록한 로그 |
| SLA | Service Level Agreement의 약자로, 서비스 수준 목표 또는 약속을 의미 |

## 2. 개발 계획

### 2.1 개발 절차 모형

#### 2.1.1 개발 절차 모형

WorkDrive는 조직 협업 도구이며, 파일 공유 정책, 권한 체계, 검색 방식, 댓글 기능 등 사용자 피드백에 따라 요구사항이 변경될 가능성이 있다. 따라서 반복적으로 기능을 구현하고 검토할 수 있는 애자일 기반 SCRUM 모델을 적용한다.

SCRUM을 적용하면 짧은 주기의 스프린트마다 우선순위가 높은 기능을 선정하여 개발하고, 리뷰와 회고를 통해 요구사항 변경과 품질 문제를 빠르게 반영할 수 있다.

#### 2.1.2 분석 및 설계 방법론

분석과 설계에는 객체지향 방법론을 적용한다. 파일, 사용자, 권한, 공유 링크, 댓글, 버전, 로그와 같은 주요 개념을 도메인 객체로 나누어 설계하면 기능 확장과 유지보수에 유리하다. 또한 권한 정책, 파일 버전 관리, 감사 로그처럼 독립적인 책임을 가진 모듈을 분리하여 테스트와 변경 관리를 쉽게 한다.

### 2.2 WBS(Work Breakdown Structure)

| 단계 | 작업 항목 | 세부 작업 | 주요 산출물 |
|------|-----------|-----------|-------------|
| 1 | 프로젝트 착수 | 주제 선정, 유사 시스템 조사, 프로젝트 범위 정의 | 프로젝트 정의서 |
| 2 | 요구사항 정의 | 사용자 요구사항, 관리자 요구사항, 품질 요구사항 정리 | 요구사항 정의서 |
| 3 | 요구사항 분석 | 기능 우선순위, 품질 목표, 권한 정책 분석 | 요구사항 분석서 |
| 4 | 설계 | 시스템 구조, DB 모델, API, 화면 흐름 설계 | 소프트웨어 설계서 |
| 5 | 구현 | 업로드/다운로드, 공유, 권한, 댓글, 검색, 버전 관리 구현 | 소스 코드 |
| 6 | 테스트 | 단위 테스트, 통합 테스트, 권한/보안 테스트, 성능 테스트 | 테스트 결과서 |

## 3. 팀 구성

### 3.1 팀 구조

본 프로젝트는 팀장, 프론트엔드, 백엔드, 데이터베이스, QA/문서 담당자가 협력하는 수평적 팀 구조를 채택한다. 팀장은 일정과 범위를 조율하지만, 주요 기능 변경과 품질 기준은 팀 전체 논의를 통해 결정한다.

과제 수행은 개인 작성 기준으로 진행하되, 실제 프로젝트 수행 시에는 인원 제한 없이 필요한 역할을 구성할 수 있다고 가정한다.

## 4. 품질 관리

### 4.1 팀 미팅 계획

#### 4.1.1 Sprint Planning Meeting

- 회의 시간: 각 스프린트 시작일
- 토의 내용: 스프린트 목표, 개발 백로그, 담당자, 완료 기준 결정
- 참석자: 팀원 전체

#### 4.1.2 Daily Scrum Meeting

- 회의 시간: 매일 10~15분
- 토의 내용: 전일 작업, 금일 작업, 장애 요소 공유
- 참석자: 팀원 전체

#### 4.1.3 Sprint Review

- 회의 시간: 각 스프린트 종료 시점
- 토의 내용: 구현 기능 시연, 요구사항 충족 여부 확인, 피드백 수집
- 참석자: 팀원 전체, 필요 시 사용자 대표

#### 4.1.4 Sprint Retrospective

- 회의 시간: 각 스프린트 종료 후
- 토의 내용: 잘한 점, 문제점, 다음 스프린트 개선 사항 도출
- 참석자: 팀원 전체

#### 4.1.5 비정기 회의

- 프로젝트 진행 중 일정 지연, 기능 변경, 보안 이슈, 품질 문제가 발생하는 경우 개최한다.
- 팀장, 담당자 또는 지도교수의 요청으로 진행할 수 있다.

### 4.2 변경 사항 관리 방법

- 기능 변경 요청은 변경 요청서 또는 GitHub Issue로 기록한다.
- 변경 요청에는 변경 사유, 영향 범위, 우선순위, 담당자, 목표 반영 시점을 포함한다.
- 권한 관리, 파일 저장, 버전 복구처럼 핵심 기능에 영향을 주는 변경은 팀 전체 검토 후 반영한다.
- 변경 전후의 요구사항, 설계, 테스트 결과는 반드시 산출물에 기록한다.
- 일정 지연이 예상될 경우 우선순위가 낮은 기능을 후순위로 조정한다.
- 변경 사항이 확정되면 관련 문서와 README를 함께 갱신한다.

### 4.3 산출물 관리 방법
